# Архитектура

## Обзор системы

Media Downloader построен на микросервисной архитектуре с четким разделением ответственности между компонентами.

## Компоненты системы

### Frontend (React + MUI)

**Технологии:**

React 18 с TypeScript для типобезопасности и современного подхода к разработке.

Material-UI v6 реализует Material Design 3 - последнюю версию дизайн-системы Google с динамическими цветами и улучшенной доступностью.

Vite используется для быстрой сборки и горячей перезагрузки модулей (HMR) во время разработки.

Axios обрабатывает HTTP запросы с автоматической сериализацией JSON и перехватчиками для обработки ошибок.

**Структура:**

```
frontend/src/
├── components/       # React компоненты
│   ├── URLInput.tsx
│   ├── MediaPreview.tsx
│   ├── FormatSelector.tsx
│   └── DownloadButton.tsx
├── services/         # API клиенты
│   └── api.ts
├── theme/            # Material Design 3 тема
│   └── theme.ts
└── App.tsx           # Главный компонент
```

**Ответственность:**

Пользовательский интерфейс - все визуальные элементы и взаимодействие с пользователем.

Валидация URL на клиенте - проверка формата URL перед отправкой на сервер для быстрой обратной связи.

Отображение информации о медиа - превью, название, длительность, доступные форматы.

Управление состоянием приложения - хранение текущего URL, выбранного формата, статуса загрузки.

### Backend (Rust + Axum)

**Технологии:**

Rust 1.83+ обеспечивает безопасность памяти без сборщика мусора и высокую производительность.

Axum - современный веб-фреймворк от создателей Tokio, построенный на tower middleware и hyper HTTP сервере.

Tokio - асинхронный runtime для неблокирующего ввода-вывода, позволяющий обрабатывать тысячи одновременных соединений.

Serde - библиотека для сериализации и десериализации данных в JSON, YAML и другие форматы.

Reqwest - HTTP клиент для взаимодействия с внешними API платформ.

**Структура:**

```
backend/src/
├── handlers/         # HTTP обработчики
│   ├── extract.rs
│   ├── download.rs
│   ├── formats.rs
│   └── mod.rs
├── extractors/       # Экстракторы платформ
│   ├── youtube.rs
│   ├── tiktok.rs
│   ├── generic.rs
│   └── mod.rs
├── services/         # Бизнес-логика
│   ├── downloader.rs
│   └── mod.rs
└── main.rs           # Entry point
```

**Ответственность:**

REST API endpoints - обработка HTTP запросов от frontend.

Извлечение метаданных - получение информации о видео/аудио с платформ.

Скачивание медиа - загрузка файлов через yt-dlp или прямые API.

Конвертация форматов - преобразование видео/аудио через FFmpeg.

Rate limiting - ограничение количества запросов для защиты от злоупотреблений.

### Reverse Proxy (Caddy)

**Функции:**

Автоматический SSL через Let's Encrypt - получение и обновление сертификатов без ручной настройки.

Проксирование запросов к backend/frontend - единая точка входа для всех сервисов.

HTTP/2 и HTTP/3 поддержка - современные протоколы для быстрой загрузки.

Статическая конфигурация через Caddyfile - простой и понятный формат настройки.

### Экстракторы платформ

Каждая платформа имеет свой экстрактор, реализующий общий интерфейс (трейт):

```rust
pub trait MediaExtractor {
    fn detect(&self, url: &str) -> bool;
    async fn extract_info(&self, url: &str) -> Result<MediaInfo>;
    async fn get_download_url(&self, url: &str, format_id: &str) -> Result<String>;
}
```

Трейт `MediaExtractor` определяет контракт, который должен выполнять каждый экстрактор. Это позволяет обрабатывать все платформы единообразно.

Метод `detect()` проверяет, подходит ли данный экстрактор для обработки URL. Например, YouTube экстрактор проверяет наличие "youtube.com" или "youtu.be" в URL.

Метод `extract_info()` - основной метод для получения метаданных. Он возвращает структуру `MediaInfo` с названием, длительностью, превью и списком форматов.

Метод `get_download_url()` генерирует прямую ссылку для скачивания в выбранном формате.

**Список экстракторов:**

YouTubeExtractor, TikTokExtractor, InstagramExtractor, FacebookExtractor, TwitterExtractor, VKExtractor, BilibiliExtractor, BlueskyExtractor, DailymotionExtractor, VimeoExtractor, RutubeExtractor, OdnoklassnikiExtractor, PinterestExtractor, NewgroundsExtractor, RedditExtractor, SoundCloudExtractor, StreamableExtractor, TumblrExtractor, TwitchExtractor, LoomExtractor, SnapchatExtractor.

GenericExtractor - универсальный экстрактор, использующий yt-dlp как fallback для платформ без специализированной реализации.

## Поток данных

### Извлечение информации

Пользователь вводит URL в поле ввода на frontend.

Frontend валидирует URL на корректность формата (проверка через regex).

Отправляется POST запрос на `/api/extract` с URL в теле запроса.

Backend получает запрос и вызывает функцию `detect_platform(url)`.

Функция `detect_platform()` проходит по списку всех экстракторов и вызывает их метод `detect()`. Первый экстрактор, вернувший `true`, будет использован.

Вызывается метод `extract_info()` выбранного экстрактора.

Экстрактор получает метаданные через API платформы или через yt-dlp subprocess.

Возвращается структура `MediaInfo` с названием, длительностью, превью и списком доступных форматов.

Frontend получает ответ и отображает превью медиа с кнопками выбора формата.

### Скачивание файла

Пользователь выбирает желаемый формат и качество из списка.

Отправляется POST запрос на `/api/download` с параметрами: URL, format_id, audio_only.

Backend создает уникальное имя файла используя UUID для предотвращения конфликтов.

Вызывается сервис `Downloader.download()` который запускает yt-dlp subprocess.

yt-dlp скачивает медиа во временную директорию `/tmp/media-downloader/`.

Если выбран audio_only режим, FFmpeg извлекает аудио дорожку и конвертирует в выбранный формат (mp3, ogg, wav).

Backend генерирует временную ссылку для скачивания вида `/downloads/UUID.ext`.

Frontend получает ответ с download_url и инициирует скачивание через браузер.

Браузер скачивает файл напрямую с сервера через ServeDir middleware.

Фоновая задача очищает временные файлы старше 1 часа для экономии дискового пространства.

## Модель данных

### MediaInfo

```rust
pub struct MediaInfo {
    pub platform: String,      // "youtube", "tiktok", etc.
    pub title: String,         // Название медиа
    pub duration: Option<u64>, // Длительность в секундах
    pub thumbnail: Option<String>, // URL превью
    pub formats: Vec<Format>,  // Доступные форматы
}
```

Структура `MediaInfo` содержит всю информацию о медиа-контенте, которую мы показываем пользователю.

Поле `platform` идентифицирует источник (youtube, tiktok, instagram и т.д.).

Поле `title` содержит оригинальное название видео или аудио.

Поле `duration` опциональное, так как некоторые платформы не предоставляют длительность или это live-трансляция.

Поле `thumbnail` содержит URL изображения для превью, если платформа его предоставляет.

Поле `formats` - вектор всех доступных форматов для скачивания (разные качества, кодеки, размеры).

### Format

```rust
pub struct Format {
    pub format_id: String,     // Уникальный ID формата
    pub quality: String,       // "720p", "1080p", "best"
    pub ext: String,           // "mp4", "webm", "mp3"
    pub filesize: Option<u64>, // Размер в байтах
    pub url: Option<String>,   // Прямая ссылка (если доступна)
}
```

Структура `Format` описывает один конкретный вариант скачивания.

Поле `format_id` - уникальный идентификатор, который мы передаем yt-dlp для выбора формата.

Поле `quality` - человекочитаемое описание качества для отображения пользователю.

Поле `ext` - расширение файла (mp4, webm, mp3, ogg и т.д.).

Поле `filesize` опциональное, так как не всегда известно заранее (особенно для live-контента).

Поле `url` содержит прямую ссылку, если платформа предоставляет её без дополнительной обработки.

## Безопасность

### Rate Limiting

Реализовано на уровне backend с использованием tower-governor middleware.

Ограничения по IP адресу:

10 запросов `/api/extract` в минуту - для предотвращения злоупотребления извлечением метаданных.

5 запросов `/api/download` в минуту - для защиты от массового скачивания.

100 запросов в час общий лимит - дополнительная защита от DDoS атак.

При превышении лимита возвращается HTTP 429 Too Many Requests.

### Валидация

**URL валидация** - проверка на корректность формата URL и отсутствие опасных символов.

**Path sanitization** - предотвращение path traversal атак (попытки доступа к файлам вне разрешенной директории).

**Size limits** - максимальный размер файла 2GB для предотвращения переполнения диска.

**Duration limits** - максимальная длительность видео 3 часа для контроля времени обработки.

### CORS

Настроен для разрешения запросов с любых источников (для удобства разработки и selfhosted использования):

```rust
let cors = CorsLayer::new()
    .allow_origin(Any)
    .allow_methods(Any)
    .allow_headers(Any);
```

В production окружении рекомендуется ограничить `allow_origin` конкретным доменом frontend.

## Масштабирование

### Горизонтальное масштабирование

Для обработки большого количества пользователей можно запустить несколько инстансов backend за load balancer.

Load Balancer (например, nginx или HAProxy) распределяет входящие запросы между backend инстансами.

Каждый backend инстанс работает независимо и может обрабатывать свою долю запросов.

Shared Storage (например, NFS или S3) используется для хранения временных файлов, доступных всем инстансам.

Это позволяет масштабироваться горизонтально, добавляя новые серверы по мере роста нагрузки.

### Кэширование

Для высоких нагрузок можно добавить Redis для кэширования метаданных.

Frontend отправляет запрос на Backend.

Backend проверяет Redis Cache на наличие закэшированных метаданных для данного URL.

При cache hit возвращаются сохраненные данные без обращения к внешним API.

При cache miss Backend обращается к Extractors, получает данные и сохраняет их в кэш.

Кэш имеет TTL (time to live) например 1 час, после чего данные обновляются.

### Очередь задач

Для асинхронной обработки тяжелых задач можно использовать RabbitMQ.

Backend получает запрос на скачивание и помещает задачу в очередь RabbitMQ.

Workers (отдельные процессы) забирают задачи из очереди и обрабатывают их.

FFmpeg запускается в Workers для конвертации видео/аудио.

Это позволяет не блокировать основной backend при длительных операциях.

## Мониторинг

### Метрики

Рекомендуется использовать Prometheus для сбора метрик и Grafana для визуализации.

Количество запросов по endpoints - отслеживание популярности функций.

Время ответа API - мониторинг производительности.

Количество успешных/неудачных скачиваний - отслеживание надежности.

Использование диска для временных файлов - предотвращение переполнения.

Использование CPU/RAM - оптимизация ресурсов.

### Логирование

Структурированное логирование через tracing библиотеку:

```rust
tracing::info!("Extracting info from URL: {}", url);
tracing::error!("Failed to download: {}", error);
```

Логи содержат контекст (URL, platform, format) для упрощения отладки.

Уровни логирования: debug, info, warn, error для фильтрации важности.

## Развертывание

### Development

В режиме разработки frontend и backend запускаются отдельно.

Frontend работает на порту 3000 с hot module replacement для быстрой разработки.

Backend работает на порту 8080 с RUST_LOG=debug для детального логирования.

Frontend напрямую обращается к Backend API.

### Production

В production все сервисы работают за Caddy reverse proxy.

Caddy слушает порты 80 (HTTP) и 443 (HTTPS).

Caddy автоматически получает SSL сертификаты от Let's Encrypt.

Frontend обслуживается как статические файлы на порту 80 внутри контейнера.

Backend работает на порту 8080 внутри контейнера.

Caddy проксирует запросы к `/api/*` на Backend, остальное на Frontend.

## Обработка ошибок

### Типы ошибок

```rust
pub enum AppError {
    InvalidUrl,
    UnsupportedPlatform,
    ExtractionFailed,
    DownloadFailed,
    ConversionFailed,
    RateLimitExceeded,
}
```

Каждый тип ошибки имеет свой HTTP статус код и сообщение для пользователя.

### Стратегия обработки

**Graceful degradation** - при ошибке специализированного экстрактора используется fallback на GenericExtractor с yt-dlp.

**Retry logic** - повторные попытки для временных сетевых ошибок (timeout, connection reset).

**User feedback** - понятные сообщения об ошибках на языке пользователя без технических деталей.

**Logging** - детальное логирование всех ошибок с контекстом для отладки разработчиками.

## Производительность

### Оптимизации

**Async/await** - все операции ввода-вывода неблокирующие благодаря Tokio runtime.

**Connection pooling** - переиспользование HTTP соединений для внешних API запросов.

**Streaming** - потоковая передача больших файлов без загрузки в память целиком.

**Compression** - Gzip сжатие для API ответов уменьшает трафик.

**CDN** - статические файлы frontend можно раздавать через CDN для быстрой загрузки.

### Benchmarks

Целевые показатели производительности:

Время извлечения метаданных меньше 2 секунд для большинства платформ.

Время начала скачивания меньше 5 секунд от нажатия кнопки.

Пропускная способность 100+ одновременных пользователей на одном инстансе.

Использование памяти меньше 512MB на инстанс в idle состоянии.
