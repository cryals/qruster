# Концепция и идеи

## Введение

Media Downloader - это универсальный selfhosted веб-сервис для скачивания аудио и видео контента с популярных платформ. Проект создан с фокусом на простоту использования, приватность и полный контроль над данными.

## Зачем это нужно?

### Проблемы существующих решений

1. **Зависимость от онлайн-сервисов**: Большинство сервисов для скачивания медиа работают онлайн, что означает:
   - Отправку ваших URL на сторонние серверы
   - Риски приватности и безопасности
   - Зависимость от доступности сервиса
   - Ограничения по количеству скачиваний

2. **Сложность установки**: Существующие selfhosted решения часто требуют:
   - Глубоких технических знаний
   - Ручной настройки множества компонентов
   - Сложного процесса обновления

3. **Ограниченная функциональность**: Многие инструменты поддерживают только YouTube или ограниченный набор платформ.

### Наше решение

Media Downloader решает эти проблемы через:

- **Полный контроль**: Все работает на вашем сервере, никакие данные не покидают вашу инфраструктуру
- **Простота развертывания**: Автоматизированные скрипты setup.sh/setup.bat делают установку тривиальной
- **Универсальность**: Поддержка 20+ платформ из коробки
- **Production-ready**: Docker, автоматический SSL, мониторинг здоровья сервисов


### Процесс скачивания

1. **Ввод URL**: Пользователь вставляет ссылку на медиа-контент
2. **Определение платформы**: Система автоматически определяет платформу по URL
3. **Извлечение метаданных**: Получение информации о видео/аудио (название, длительность, доступные форматы)
4. **Выбор формата**: Пользователь выбирает желаемый формат и качество
5. **Скачивание**: Backend скачивает контент во временную директорию
6. **Конвертация** (опционально): FFmpeg конвертирует в нужный формат
7. **Отдача файла**: Пользователь получает готовый файл
8. **Очистка**: Временные файлы удаляются

## Принципы работы экстракторов

### Архитектура экстракторов

Каждая платформа имеет свой экстрактор, реализующий общий trait:

```rust
pub trait MediaExtractor {
    // Определяет, подходит ли этот экстрактор для данного URL
    fn detect(&self, url: &str) -> bool;
    
    // Извлекает информацию о медиа
    async fn extract_info(&self, url: &str) -> Result<MediaInfo>;
    
    // Получает прямые ссылки для скачивания
    async fn get_download_urls(&self, url: &str, quality: Quality) 
        -> Result<Vec<DownloadOption>>;
}
```

### Стратегии извлечения

1. **Официальные API**: Для платформ с публичными API (YouTube, SoundCloud)
2. **Парсинг веб-страниц**: Извлечение данных из HTML/JavaScript
3. **Обратная инженерия**: Анализ мобильных API и внутренних endpoints
4. **Fallback на yt-dlp**: Универсальный инструмент как запасной вариант

### Обработка различных сценариев

- **Видео с аудио**: Прямое скачивание или объединение потоков
- **Видео без аудио**: Скачивание только видео потока
- **Только аудио**: Извлечение аудио дорожки или скачивание аудио формата
- **Плейлисты**: Пакетное скачивание (будущая функция)
- **Приватный контент**: Поддержка аутентификации (будущая функция)

## Безопасность и rate limiting

### Меры безопасности

1. **Валидация URL**: Проверка на валидность и безопасность URL
2. **Ограничение размера**: Максимальный размер скачиваемых файлов
3. **Sanitization путей**: Предотвращение path traversal атак
4. **CORS настройки**: Ограничение доступа к API
5. **Rate limiting**: Ограничение количества запросов

### Rate Limiting стратегия

```rust
// Ограничения по IP адресу
- 10 запросов на извлечение информации в минуту
- 5 скачиваний в минуту
- 100 запросов в час

// Ограничения по размеру
- Максимальный размер файла: 2GB
- Максимальная длительность видео: 3 часа
```

### Защита от злоупотреблений

- Временные файлы автоматически удаляются через 1 час
- Ссылки для скачивания истекают через 30 минут
- Логирование всех запросов для аудита
- Опциональная аутентификация (будущая функция)

## Технологические решения

### Почему Rust для backend?

1. **Производительность**: Близко к C/C++, критично для обработки медиа
2. **Безопасность памяти**: Отсутствие segfaults и data races
3. **Конкурентность**: Tokio для эффективной асинхронной обработки
4. **Экосистема**: Отличные библиотеки (Axum, reqwest, serde)
5. **Размер бинарника**: Маленькие Docker образы

### Почему React + MUI?

1. **Material Design 3**: Официальная реализация от Google
2. **Компонентный подход**: Переиспользуемые UI элементы
3. **TypeScript**: Типобезопасность на frontend
4. **Экосистема**: Огромное количество библиотек
5. **Производительность**: Virtual DOM и оптимизации

### Почему Docker?

1. **Изоляция**: Все зависимости в контейнере
2. **Воспроизводимость**: Одинаковое окружение везде
3. **Простота развертывания**: docker-compose up
4. **Масштабируемость**: Легко добавить больше инстансов
5. **Обновления**: Простой rollback при проблемах

### Почему Caddy?

1. **Автоматический SSL**: Let's Encrypt из коробки
2. **Простая конфигурация**: Понятный Caddyfile
3. **HTTP/2 и HTTP/3**: Современные протоколы
4. **Reverse proxy**: Проксирование на backend/frontend
5. **Безопасность**: Secure defaults
